Understanding the Java Exception Hierarchy

In Java, an Exception is an event that disrupts the normal flow of a program. All exception types are subclasses of the Throwable class. This hierarchy is divided into two main branches: Checked Exceptions and Unchecked Exceptions (RuntimeExceptions).

1. Checked Exceptions (Compile-Time Exceptions)

These are exceptions that the compiler forces you to handle (using try-catch or throws). They represent external errors that a well-written application should anticipate and recover from.

  • SQLException: Database access errors (connection failure, query syntax issues).
  • IOException: General Input/Output failures (e.g., disk full, network down).
  • FileNotFoundException: Attempting to open a file that does not exist.
  • ClassNotFoundException: The JVM cannot find a specific class at runtime.
  • InterruptedException: A thread was interrupted while sleeping or waiting.
  • ParseException: Errors during data parsing (e.g., converting a string to a date).

2. Unchecked Exceptions (Runtime Exceptions)

These extend RuntimeException. The compiler does not force you to handle them. They usually indicate programming bugs, such as logic errors or improper use of an API. They can be prevented with better coding practices.

  • NullPointerException: Accessing a method or field on a null object.
  • ArithmeticException: Dividing an integer by zero.
  • ArrayIndexOutOfBoundsException: Accessing an array with an invalid index.
  • IllegalArgumentException: Passing an illegal or inappropriate argument to a method.
  • NumberFormatException: Trying to convert a non-numeric string to a number.
  • ClassCastException: Trying to cast an object to a class it does not belong to.
  • ConcurrentModificationException: Modifying a collection while iterating over it.
  • IllegalStateException: Calling a method when the object is in an invalid state.

Best Practices

  • Handle Checked Exceptions: Always handle them gracefully to prevent application crashes.
  • Prevent Unchecked Exceptions: Use defensive checks (e.g., if (object != null)) to avoid them.
  • Specific over Generic: Catch specific exceptions (like IOException) rather than catching the generic Exception.
  • Log and Re-throw: In service layers, log the exception and re-throw a custom business exception if needed.

Practical Example

try {
    FileReader file = new FileReader("data.txt");
    BufferedReader br = new BufferedReader(file);
    String line = br.readLine();
} catch (FileNotFoundException e) {
    System.err.println("File not found! Creating a default one...");
} catch (IOException e) {
    e.printStackTrace();
}
