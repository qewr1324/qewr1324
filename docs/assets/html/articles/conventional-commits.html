چرا Conventional Commits؟

استاندارد Conventional Commits یک قرارداد برای ساختار پیام‌های کامیت است. این استاندارد به تیم‌های توسعه کمک می‌کند تا تاریخچه‌ی پروژه را بهتر بفهمند و ابزارهایی مانند semantic-release بتوانند به طور خودکار نسخه‌دهی (Versioning) را مدیریت کنند.

ساختار اصلی یک کامیت

<type>(<scope>): <subject>

<body>

<footer>

انواع اصلی (Types)

  • feat: یک ویژگی جدید به برنامه اضافه می‌شود.
  • fix: رفع یک باگ در برنامه.
  • docs: تغییرات مربوط به مستندات.
  • style: تغییراتی که روی منطق برنامه تأثیر نمی‌گذارند (فرمت‌بندی، فاصله‌ها و...).
  • refactor: بازنویسی کد بدون تغییر عملکرد ظاهری.
  • test: اضافه کردن یا اصلاح تست‌ها.
  • chore: وظایف نگهداری پروژه (به‌روزرسانی پکیج‌ها، تنظیمات CI/CD).

مثال عملی

feat(auth): add two-factor authentication
Implement TOTP-based F2A for user login security.
Closes #456

نکات کلیدی

  • فاعل امری (Imperative Mood): مثل add به جای added.
  • کوتاه بودن: Subject نباید بیشتر از ۵۰ کاراکتر باشد.
  • چرا نه چگونه: در بخش Body توضیح دهید چرا این تغییر انجام شده، نه چگونه.
